## The main thread is JavaScript's single execution lane

Browsers run JavaScript, layout, and painting on a single main thread. When any JavaScript call takes a long time to return, it holds up everything else — animations stutter, clicks go unresponsive, and the browser may show a "page not responding" warning. This is called **jank**.

## Synchronous storage: Web Storage and cookies

`localStorage`, `sessionStorage`, and `document.cookie` are all **synchronous**. When you call `localStorage.getItem('key')`, the browser reads from disk, returns the value, and only then lets your code continue. Your script cannot do anything else while it waits.

For a single small read this is imperceptible. But problems surface when you:

- Loop over thousands of keys
- Store large serialised objects (many kilobytes each)
- Write on every keystroke or scroll event
- Parse the entire cookie string on a hot code path

Because all of this happens on the main thread, heavy Web Storage use can introduce measurable frame-time delays.

```js
// Every one of these calls is synchronous — the thread blocks until each completes
const start = performance.now();
for (let i = 0; i < 1000; i++) {
  localStorage.setItem('key-' + i, 'value-' + i);
}
const elapsed = performance.now() - start;
console.log('1 000 writes took ' + elapsed.toFixed(2) + ' ms');
```

## Asynchronous storage: IndexedDB and Cache API

IndexedDB and the Cache API are **asynchronous**. Every operation returns a Promise (or fires an event) and your code continues immediately. The actual I/O happens off the main thread. When results are ready, the browser calls your callback or resolves your Promise.

```js
// This returns immediately — the write happens in the background
const db = await idb.openDB('my-db', 1, {
  upgrade(db) { db.createObjectStore('items'); }
});
await db.put('items', { name: 'Alice' }, 'user-1');
console.log('Write complete — main thread was never blocked');
```

## Runnable timing comparison

The snippet below times 10 000 `localStorage.setItem` calls synchronously, logs the elapsed time, and then cleans up every `demo:` key it created.

export const timingCode = `const COUNT = 10_000;
const start = performance.now();

// Synchronous: all 10 000 writes block the thread
for (let i = 0; i < COUNT; i++) {
  localStorage.setItem('demo:bench-' + i, 'v' + i);
}
const elapsed = performance.now() - start;
console.log(COUNT + ' synchronous localStorage writes: ' + elapsed.toFixed(2) + ' ms');

// Clean up all demo: keys
let removed = 0;
for (let i = 0; i < COUNT; i++) {
  localStorage.removeItem('demo:bench-' + i);
  removed++;
}
console.log('Cleaned up ' + removed + ' demo keys');
console.log('Remaining demo keys in localStorage:',
  Object.keys(localStorage).filter(k => k.startsWith('demo:')).length);`;

<Callout title="Performance tip">
If your storage access pattern involves large objects, frequent writes, or happens inside a scroll or animation callback, switch to IndexedDB. Its async, off-thread model keeps the main thread free for rendering and input handling, eliminating the jank that synchronous Web Storage can introduce.
</Callout>
