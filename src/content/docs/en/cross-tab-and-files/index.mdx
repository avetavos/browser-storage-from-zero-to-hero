## Coordinating across tabs and persisting files

Modern web apps often run in multiple browser tabs simultaneously — a user may have the same dashboard open in two windows. When one tab updates data, how does the other find out? When the app needs to write a large file, how does it avoid clobbering work in a concurrent tab? This module answers both questions.

The browser gives you four complementary APIs:

- **BroadcastChannel** — publish/subscribe messaging across every tab (and worker) on the same origin.
- **`storage` event** — fired on other tabs when `localStorage` is written; the original notification mechanism before BroadcastChannel.
- **Web Locks** — a mutual-exclusion primitive that serialises access to a named resource, preventing race conditions.
- **File System Access / OPFS** — read and write real files (or a private origin sandbox) with a streaming, async API.

## Quick comparison

| API | Delivery | Requires gesture | Works in workers | Persistent |
|---|---|---|---|---|
| BroadcastChannel | other tabs | no | yes | no |
| `storage` event | other tabs | no | no | no |
| Web Locks | same tab | no | yes | no |
| File System Access | — | yes (picker) | no | yes |
| OPFS | — | no | yes | yes |

## Your first cross-tab message

The snippet below creates a BroadcastChannel and immediately posts and receives a message in the same page. In a real app the sender and receiver are in separate tabs.

export const ctIndexCode = `// Both sender and receiver use the same channel name.
const ch = new BroadcastChannel('demo:intro');

// Listen for messages (in this demo: same-tab echo)
ch.onmessage = (e) => {
  console.log('Received:', JSON.stringify(e.data));
  ch.close();
};

// Send a message — in another tab this would arrive via onmessage there
ch.postMessage({ type: 'PING', ts: Date.now() });
console.log('Message posted. Waiting for echo...');
// NOTE: cross-tab delivery requires a real second tab open on the same origin.`;

<Callout title="Key point">
BroadcastChannel messages do NOT fire in the tab that called `postMessage` — only in other tabs and workers sharing the same channel name. The snippet above receives its own message because both sender and receiver are in the same script; in production they live in separate tabs.
</Callout>
