## คุกกี้คืออะไร?

คุกกี้คือสตริงขนาดเล็ก — แต่ละชิ้นมีขนาดไม่เกินประมาณ 4 KB — ที่เบราว์เซอร์จัดเก็บแยกตามโดเมนและ**ส่งไปพร้อมกับ HTTP request ทุกครั้งโดยอัตโนมัติ** นี่คือสิ่งที่ทำให้คุกกี้แตกต่างจากกลไกจัดเก็บข้อมูลอื่นๆ ในเบราว์เซอร์ทั้งหมด

เมื่อ server ส่ง `Set-Cookie` response header มา เบราว์เซอร์จะเก็บคุกกี้นั้นและส่งกลับไปใน request ถัดๆ ไปที่ตรงกับโดเมนนั้น (ขึ้นอยู่กับ attribute ของคุกกี้ เช่น `Path`, `Domain`, `Secure`, และ `SameSite`)

เมื่อ JavaScript เรียก `document.cookie = 'name=value'` มันจะตั้งค่าคุกกี้ฝั่ง client — แต่เบราว์เซอร์จะยังคงส่งคุกกี้นั้นไปยัง server ใน request ถัดไปอีกด้วย

## คุกกี้ที่ตั้งฝั่ง client vs ฝั่ง server

มีสองวิธีที่คุกกี้จะถูกเก็บไว้ใน cookie jar ของเบราว์เซอร์:

**ตั้งโดย server (ผ่าน `Set-Cookie` header):**

```http
HTTP/1.1 200 OK
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
```

Server เป็นผู้ควบคุมค่าและ attribute ของคุกกี้ คุกกี้ที่มี `HttpOnly` จะมองไม่เห็นจาก JavaScript เลย — `document.cookie` จะไม่แสดงมันออกมาไม่ว่ากรณีใด

**ตั้งโดย client (ผ่าน `document.cookie`):**

```js
document.cookie = 'theme=dark; Max-Age=86400; SameSite=Lax';
```

JavaScript สามารถตั้งค่า อ่าน และลบคุกกี้ที่ไม่ได้ทำเครื่องหมาย `HttpOnly` ได้

## คุกกี้ vs Web Storage: เลือกใช้เมื่อไหร่

คำถามสำคัญที่สุดคือ: **server จำเป็นต้องรับค่านี้หรือเปล่า?**

| กรณีการใช้งาน | ตัวเลือกที่ดีที่สุด | เหตุผล |
|---|---|---|
| Auth token / session ID | Cookie (HttpOnly + Secure) | Server ต้องการมัน; HttpOnly ป้องกันการขโมยผ่าน XSS |
| CSRF token | Cookie | Server ตั้งค่า; JS อ่านเพื่อใส่ใน request body |
| ค่าความชอบของผู้ใช้ (ธีม, ภาษา) | Web Storage | ใช้ฝั่ง client เท่านั้น; ไม่จำเป็นต้องส่งไป server ทุก request |
| ตะกร้าสินค้า (client-only) | Web Storage หรือ IndexedDB | ข้อมูลมากกว่า รูปแบบสมบูรณ์กว่า; ไม่มีภาระทางเครือข่าย |
| Feature flags จาก server | Cookie | Server ปรับ response ได้ก่อน JS โหลด |

กฎพื้นฐาน: **ใช้คุกกี้สำหรับทุกสิ่งที่ server ต้องรับ; ใช้ Web Storage สำหรับ UI state ที่เป็น client-only**
